# Plan: architecture cleanup for clarity, state-driven flow, and future extensibility

## 1. Architectural charter

### Objective
Make NeoZeus explicitly **state-driven**.

Architecture target:
- use cases mutate **authoritative state only**
- adapters perform external I/O only
- UI reads **UI view-models** only
- rendering syncs **render projections** only
- orchestration never directly mutates presentation entities

### Core principle
The app should be understandable as:

```text
Input -> AppCommand -> Use-case handler -> Authoritative state
                                      -> Adapter calls
Authoritative state -> UI view-models -> widgets
Authoritative state -> render projections -> rendering
```

### Hard dependency rules
- UI/widgets depend on view-models + command types only.
- Use-case handlers may depend on authoritative state + adapters.
- Domain state must not depend on HUD/rendering/ECS presentation details.
- Adapters must not own product policy.
- Render projection systems must not own product decisions.
- New messaging APIs must never accept `TerminalId`.
- Orchestration code must not spawn/despawn panel/frame/widget render entities directly.

### Non-goals
- no daemon/runtime rewrite
- no plugin framework
- no dynamic trait-heavy widget system
- no network/distributed architecture yet

### Traps
- “service layer” sprawl: avoid broad `FooService` buckets that become hidden monoliths
- registry overreach: do not build a mini UI framework inside Bevy
- projection leakage: do not let use cases clean up sprite/camera/entity state

## 2. Authoritative state model

Only four authoritative stores should exist at the app/domain level.

### 2.1 `AgentCatalog`
Agent metadata only.

Owns:
- `AgentId`
- label
- kind
- capabilities
- persistent metadata

Does **not** own:
- active selection
- terminal snapshots
- widget state
- render state

```rust
pub(crate) struct AgentCatalog {
    pub(crate) agents: BTreeMap<AgentId, AgentRecord>,
    pub(crate) order: Vec<AgentId>,
}

pub(crate) struct AgentRecord {
    pub(crate) label: String,
    pub(crate) kind: AgentKind,
    pub(crate) capabilities: AgentCapabilities,
}
```

### 2.2 `AgentRuntimeIndex`
Links agents to runtime terminals/sessions.

Owns:
- `AgentId <-> TerminalId`
- session name linkage
- runtime attachment/liveness linkage

Does **not** own:
- selection order
- widget-facing view state
- render entities

```rust
pub(crate) struct AgentRuntimeIndex {
    pub(crate) agent_to_runtime: BTreeMap<AgentId, AgentRuntimeLink>,
    pub(crate) terminal_to_agent: BTreeMap<TerminalId, AgentId>,
}

pub(crate) struct AgentRuntimeLink {
    pub(crate) primary_terminal: Option<TerminalId>,
    pub(crate) session_name: Option<String>,
    pub(crate) lifecycle: AgentRuntimeLifecycle,
}
```

### 2.3 `ConversationStore`
Messaging/task/message delivery domain state.

Owns:
- conversations
- messages
- delivery state
- conversation membership
- optional task/domain records if tasks stay coupled to agent conversations

Does **not** own:
- transport implementation
- composer UI state
- widget-local cached rows

```rust
pub(crate) struct ConversationStore {
    pub(crate) conversations: BTreeMap<ConversationId, ConversationRecord>,
    pub(crate) messages: BTreeMap<MessageId, MessageRecord>,
}
```

### 2.4 `AppSessionState`
Everything session-local that is not core domain truth.

Owns:
- active agent
- inspect/navigation selection state
- visibility/focus mode
- widget shells/layout/z-order
- composer state
- temporary input capture/session modes

Does **not** own:
- persistent agent metadata
- message history
- runtime snapshots
- render projections

```rust
pub(crate) struct AppSessionState {
    pub(crate) active_agent: Option<AgentId>,
    pub(crate) visibility_mode: VisibilityMode,
    pub(crate) widget_layout: WidgetLayoutState,
    pub(crate) composer: ComposerState,
    pub(crate) direct_input_terminal: Option<TerminalId>,
}
```

### Invariants
- `AgentId` is the product identity.
- `TerminalId` stays runtime/presentation identity only.
- selection state belongs to `AppSessionState`, not the agent domain.
- widget shell state belongs to `AppSessionState`, not widget code.

### Traps
- do not recreate a giant `AgentRegistry` monolith
- do not leave labels in both `AgentDirectory` and `AgentCatalog`
- do not persist the same concept from multiple owners

## 3. Command model

### Goal
Replace HUD/request fanout with one app-level command surface, but keep it structured.

### Shape

```rust
pub(crate) enum AppCommand {
    Agent(AgentCommand),
    Terminal(TerminalCommand),
    Conversation(ConversationCommand),
    Composer(ComposerCommand),
    Widget(WidgetCommand),
}
```

#### Agent commands
```rust
pub(crate) enum AgentCommand {
    SpawnTerminal,
    SpawnShellTerminal,
    Focus(AgentId),
    KillActive,
    Rename { agent_id: AgentId, label: String },
}
```

#### Terminal commands
```rust
pub(crate) enum TerminalCommand {
    SendCommand { target: TerminalId, command: String },
    ResetActiveView,
    ToggleActiveDisplayMode,
}
```

#### Conversation commands
```rust
pub(crate) enum ConversationCommand {
    SendMessage {
        conversation_id: ConversationId,
        sender: AgentId,
        body: String,
    },
    ClearDoneTasks { agent_id: AgentId },
    ConsumeNextTask { agent_id: AgentId },
}
```

#### Composer commands
```rust
pub(crate) enum ComposerCommand {
    Open(ComposerRequest),
    Submit,
    Cancel,
}
```

#### Widget commands
```rust
pub(crate) enum WidgetCommand {
    Toggle(HudWidgetKey),
    Reset(HudWidgetKey),
}
```

### Rule
Commands describe intent. They are not events, not effects, not internal work queues.

### Migration
1. translate current HUD/input/verification outputs into `AppCommand`
2. add temporary adapter from `AppCommand` to old request enums
3. migrate handlers one flow at a time
4. delete old request enums once parity is reached

### Invariants
- one mutation entry point from UI/input
- explicit ordering remains in schedule
- existing shortcuts/flows remain behavior-compatible during migration

### Traps
- do not keep permanent `AppCommand -> old request fanout`
- do not flatten all commands into one giant enum without subtypes

## 4. Use-case handlers

### Goal
Use explicit use cases instead of broad service buckets.

### Required handlers
- `restore_app`
- `spawn_agent_terminal`
- `focus_agent`
- `kill_active_agent`
- `open_composer`
- `submit_composer`
- `toggle_widget`
- `reset_widget`
- `send_message`
- `send_terminal_command`

### Recommended layout
- `src/app/use_cases/restore_app.rs`
- `src/app/use_cases/spawn_agent_terminal.rs`
- `src/app/use_cases/focus_agent.rs`
- `src/app/use_cases/kill_active_agent.rs`
- `src/app/use_cases/open_composer.rs`
- `src/app/use_cases/submit_composer.rs`
- `src/app/use_cases/toggle_widget.rs`

Shared helpers can live under `src/app/support/*`.

### Rule
Use-case handlers may:
- read/write authoritative state
- call adapters
- decide application policy/order

They may **not**:
- manipulate presentation ECS entities directly
- own widget rendering logic
- own terminal raster/projection logic

### Example: `kill_active_agent`

#### Bad
```rust
fn kill_active_agent(...) {
    // kill runtime
    // clear focus
    // remove labels
    // despawn panel entity
    // despawn frame entity
    // update hud visibility
}
```

#### Correct
```rust
fn kill_active_agent(...) -> Result<(), String> {
    let agent_id = app_session.active_agent.ok_or("no active agent")?;
    let terminal_id = runtime_index
        .agent_to_runtime
        .get(&agent_id)
        .and_then(|link| link.primary_terminal);

    if let Some(terminal_id) = terminal_id {
        runtime_adapter.kill_terminal(terminal_id)?;
    }

    runtime_index.mark_stopped(agent_id);
    app_session.select_replacement_agent(&catalog, &runtime_index);
    Ok(())
}
```

Projection cleanup happens later in projection sync.

### Invariants
- use-case handlers are the only place where product policy lives
- handlers remain narrow and named by user-visible intent
- startup restore is a use case, not bootstrap sludge

### Traps
- do not reintroduce a broad `TerminalService` that owns app policy + projections + persistence
- do not move “just one cleanup” back into orchestration code; that reopens the leak

## 5. Adapter boundaries

Adapters perform external I/O only.

### Required adapters
- `TerminalRuntimeAdapter`
- `PersistenceAdapter`
- `MessageTransportAdapter`

### 5.1 `TerminalRuntimeAdapter`
Wraps daemon/runtime attach/spawn/kill/send operations.

Owns:
- session spawn/attach/kill/send/resize I/O

Does **not** own:
- active agent policy
- widget logic
- conversation state
- UI-specific selection behavior

### 5.2 `PersistenceAdapter`
Owns serialization/deserialization only.

Should support:
- agents/catalog persistence
- app session persistence where relevant
- conversation persistence if in scope
- migration/version upgrade paths

### 5.3 `MessageTransportAdapter`
Owns message delivery transport only.

```rust
pub(crate) trait MessageTransportAdapter {
    fn send_agent_message(
        &self,
        sender: AgentId,
        recipients: &[AgentId],
        body: &str,
    ) -> Result<TransportReceipt, String>;
}
```

Initial implementation may route via the current local terminal mechanism or a Zeus message path, but that leak must stay inside this adapter only.

### Invariants
- transport failure updates conversation/message delivery state
- adapter errors are explicit; no silent swallowing
- adapters do not mutate product policy on their own

### Traps
- do not let UI/widgets call adapters directly
- do not let persistence own reconciliation policy

---

## 6. UI view-models

### Goal
Widgets read derived UI data, not raw stores/managers.

### Naming rule
Use **UI view-models** for widget-facing derived state. Do not call them projections.

### Required view-models
- `AgentListView`
- `ConversationListView`
- `ComposerView`
- optional `WidgetChromeView`

### Example
```rust
pub(crate) struct AgentListView {
    pub(crate) items: Vec<AgentListItemView>,
}

pub(crate) struct AgentListItemView {
    pub(crate) agent_id: AgentId,
    pub(crate) label: String,
    pub(crate) focused: bool,
    pub(crate) runtime_state: AgentRuntimeBadge,
    pub(crate) unread_count: u32,
}
```

### Source inputs
View-model builders may read:
- `AgentCatalog`
- `AgentRuntimeIndex`
- `ConversationStore`
- `AppSessionState`

Widgets should not read those raw stores directly.

### Why
- smaller widget code
- better tests
- less subsystem coupling
- easier future features like unread badges or offline agents

### Invariants
- widgets own no domain truth
- row identity is `AgentId`
- messaging widgets never depend on `TerminalId`

### Traps
- do not create persistent caches here unless needed; view-models are derived
- do not let widgets keep their own copy of conversation state

---

## 7. Render projections

### Goal
Keep ECS/render synchronization derived and reactive.

### Naming rule
Use **render projections** for ECS/render derived state. Do not call them view-models.

### Required projection sync areas
- terminal panel/frame projection
- HUD shell projection
- widget effect projection

### Rule
Projection sync reads authoritative state and reconciles ECS entities.

It may:
- spawn/despawn presentation entities
- update transforms/sprites/visibility
- update offscreen effect entities/resources

It may **not**:
- decide product behavior
- change conversations/agent metadata
- decide active selection policy

### Example structure
```rust
fn sync_terminal_render_projection(
    catalog: Res<AgentCatalog>,
    runtime_index: Res<AgentRuntimeIndex>,
    app_session: Res<AppSessionState>,
    // render resources/queries...
) {
    // converge terminal panel/frame entities from state
}
```

### Critical rule
All current direct presentation mutations embedded in lifecycle helpers must move here.

### Invariants
- projections are disposable; authoritative state is elsewhere
- startup restore restores state only; projection sync converges visuals later
- terminal presentation split remains intact

### Traps
- do not mix widget-facing view-model logic into render projection code
- do not let render projection code become a place for product policy hacks

---

## 8. Widget architecture

### Goal
Make widgets cheaper to add without inventing a widget framework.

### Principle
Keep widget behavior in ordinary systems. Registry is metadata only.

### Registry scope
Registry should contain only:
- stable widget key
- display title
- placement policy
- default shell
- shortcut
- persistence key if needed

```rust
pub(crate) struct HudWidgetDefinition {
    pub(crate) key: HudWidgetKey,
    pub(crate) title: &'static str,
    pub(crate) placement: HudWidgetPlacement,
    pub(crate) default_shell: HudWidgetShell,
    pub(crate) shortcut: Option<u8>,
}
```

### Generic shell state
Shell/layout belongs in `AppSessionState`:
- enabled
- rect
- alpha
- z-order
- draggable
- placement

### Placement policy before registry expansion
Introduce generic placement first:

```rust
pub(crate) enum HudWidgetPlacement {
    Floating,
    DockedLeft,
    DockedRight,
    FullHeightLeft,
    FixedModal,
}
```

Then use the registry for identity + defaults.

### Migration order
1. generic shell/placement state
2. narrow widget registry
3. migrate existing widgets (`AgentList`, `DebugToolbar`)
4. add conversation widgets later

### Invariants
- no widget callback mini-framework
- widget-local state stays in normal resources/components
- persistence keys are stable strings, not display titles

### Traps
- do not hide widget behavior in registry metadata
- do not keep a giant closed-world `HudModuleModel` enum after migration

## 9. Composer architecture

### Goal
Replace message box/task dialog with one reusable composer.

### Ownership
Composer belongs to `AppSessionState` as session/UI state.

Submitted results mutate domain state; drafts do not become domain truth unless explicitly persisted later.

### Shape
```rust
pub(crate) enum ComposerMode {
    TerminalCommand { target: TerminalId },
    AgentMessage {
        conversation_id: ConversationId,
        sender: AgentId,
        recipients: Vec<AgentId>,
    },
    TaskEdit { agent_id: AgentId },
    AgentRename { agent_id: AgentId },
}

pub(crate) struct ComposerState {
    pub(crate) active: Option<ComposerSession>,
}
```

### Key rule
Draft identity is keyed by logical target identity, not modal type.

Examples:
- terminal command draft -> `TerminalId`
- agent message draft -> `(ConversationId, sender, recipients)`
- rename draft -> `AgentId`

### Submission path
```rust
fn submit_composer(session: &ComposerSession, body: String) -> AppCommand {
    match &session.mode {
        ComposerMode::TerminalCommand { target } => AppCommand::Terminal(
            TerminalCommand::SendCommand { target: *target, command: body }
        ),
        ComposerMode::AgentMessage { conversation_id, sender, .. } => AppCommand::Conversation(
            ConversationCommand::SendMessage {
                conversation_id: *conversation_id,
                sender: *sender,
                body,
            }
        ),
        ComposerMode::TaskEdit { agent_id } => AppCommand::Conversation(
            ConversationCommand::ClearDoneTasks { agent_id: *agent_id }
        ),
        ComposerMode::AgentRename { agent_id } => AppCommand::Agent(
            AgentCommand::Rename { agent_id: *agent_id, label: body }
        ),
    }
}
```

### Invariants
- keyboard precedence stays: composer > direct terminal input > normal shortcuts
- existing editor behavior/kill-ring/drafts must not regress during migration
- composer state is UI/session state, not conversation history

### Traps
- do not keep separate modal-specific input stacks after composer migration
- do not let composer submit directly to transport adapters

## 10. Startup / restore architecture

### Goal
Make restore a use case that rebuilds authoritative state only.

### `restore_app` responsibilities
- load persisted agent/session/conversation state
- query live daemon/runtime state
- reconcile persisted and live runtime linkage
- restore `AgentCatalog`
- restore `AgentRuntimeIndex`
- restore `ConversationStore`
- initialize `AppSessionState`

### It should not
- perform widget render setup logic
- manually spawn presentation entities
- own post-restore rendering policies beyond state seeding

### Why
If startup restores only state, normal view-model and projection sync systems will converge the rest automatically. That removes one major cross-layer blob.

### Invariants
- persisted labels/linkage survive migration
- verifier/scenario startup remains a separate explicit entry path
- startup ordering remains deterministic

### Traps
- do not keep half the restore flow in `startup.rs` and half in the new use case
- decide persistence ownership before migrating labels/linkage

---

## 11. Persistence / versioning

### Decision points that must be settled before migration lands
- label source of truth: `AgentCatalog`
- agent<->terminal linkage source of truth: `AgentRuntimeIndex`
- widget shell persistence owner: `AppSessionState`
- composer drafts persisted or in-memory only
- conversation/message persistence in scope now or follow-up

### Rules
- one owner per persisted concept
- version bump on identity migration from terminal-keyed to agent-keyed records
- migration readers must preserve existing session data where possible

### Invariants
- no split-brain label ownership
- no duplicate linkage ownership
- no data loss on restore of existing user sessions

### Traps
- if both old terminal session persistence and new agent persistence store labels, the architecture stays dirty

## 12. Schedule / system topology

### Goal
Preserve explicit ordering while aligning it with the new architecture.

### Recommended final shape
- `UiInput`
- `HudInput`
- `AppCommandDispatch`
- `UseCaseExecution`
- `UiViewModelSync`
- `TerminalPoll`
- `TerminalRaster`
- `RenderProjectionSync`
- `HudAnimation`
- `HudRender`
- `Redraw`

### Notes
- `AppCommandDispatch`: translate Bevy input/widget output into `AppCommand`
- `UseCaseExecution`: mutate authoritative state + call adapters
- `UiViewModelSync`: build widget-facing derived state
- `RenderProjectionSync`: converge ECS/render entities from authoritative state

### Rule
`app/schedule.rs` should wire coarse bundles/sets, not remain a permanent catalog of every low-level leaf system.

### Invariants
- terminal poll/raster ordering remains correct
- no use-case mutation after projection/render in the same frame unless intentional
- redraw behavior remains deterministic

### Traps
- do not merge view-model sync and render projection sync; they solve different problems

## 13. Migration order

### Phase A — freeze behavior
Add characterization tests for:
- spawn/focus/kill behavior
- inspect navigation
- startup restore/reconcile
- direct-input precedence
- message/task drafting behavior
- visibility/focus transitions

### Phase B — command model
- add `AppCommand`
- translate current HUD/input/verification output into `AppCommand`
- add temporary adapter into old request system

### Phase C — state split
- add `AgentCatalog`
- add `AgentRuntimeIndex`
- add `AppSessionState`
- move labels/linkage/selection/widget-shell state to new owners

### Phase D — restore use case
- implement `restore_app`
- move startup reconciliation into it
- leave rendering convergence to projection sync

### Phase E — composer
- extract editor state
- add `ComposerState`
- reimplement message box/task dialog flows on composer

### Phase F — conversations
- add `ConversationStore`
- add `MessageTransportAdapter`
- route composer submission through conversation commands/use cases

### Phase G — widget cleanup
- add generic shell/placement state
- add narrow widget registry
- migrate `AgentList` and `DebugToolbar`

### Phase H — projections and cleanup
- move direct presentation cleanup out of orchestration code
- finalize render projection sync ownership
- remove old request enums, `AgentDirectory`, old modal types, and closed-world widget enums

### Rule
Do not migrate 3 layers at once. Keep each phase narrowly scoped and behavior-preserving.

## 14. First widgets/features to prove the architecture

### First migrated widget
`AgentList`

#### Requirements
- rows keyed by `AgentId`
- label/status from `AgentCatalog` + `AgentRuntimeIndex`
- focus via `AppCommand::Agent(AgentCommand::Focus(...))`
- future unread/status badges supported naturally

### First new feature
conversation list + thread pane

#### Requirements
- conversation state read from `ConversationStore`
- send via composer -> conversation command -> use-case handler -> transport adapter
- no widget-local message ownership

### Why these two
If these work cleanly, the architecture is genuinely improved. If they require bypasses, the plan is still wrong.

## 15. Hard invariants / forbidden dependencies

### Must remain true
- terminal domain stays separate from presentation domain
- daemon stays runtime/session authority
- `AgentId` is product identity
- widgets do not own product domain state
- UI view-models are derived only
- render projections are derived only
- transport/delivery failures are explicit
- restore/persistence must not lose labels/linkage

### Forbidden dependencies
- domain modules importing HUD/rendering types
- widgets directly reading runtime adapters
- widgets directly reading raw domain stores after view-model migration
- messaging APIs taking `TerminalId`
- use-case handlers mutating render ECS entities directly
- registry-based widget behavior callbacks becoming the new architecture

## 16. Most important early code-shape changes

### Authoritative state split
```rust
pub(crate) struct AgentCatalog { /* metadata */ }
pub(crate) struct AgentRuntimeIndex { /* AgentId <-> TerminalId/session linkage */ }
pub(crate) struct ConversationStore { /* conversations/messages/delivery */ }
pub(crate) struct AppSessionState { /* active agent, widget shells, composer */ }
```

### Structured command model
```rust
pub(crate) enum AppCommand {
    Agent(AgentCommand),
    Terminal(TerminalCommand),
    Conversation(ConversationCommand),
    Composer(ComposerCommand),
    Widget(WidgetCommand),
}
```

### Narrow widget registry
```rust
pub(crate) const HUD_WIDGETS: &[HudWidgetDefinition] = &[
    AGENT_LIST_WIDGET,
    DEBUG_TOOLBAR_WIDGET,
];
```

### State-driven render convergence
```rust
fn sync_render_projections_from_state(...) {
    // spawn/update/despawn render entities from authoritative state
}
```

### Final rule
If a proposed change cannot be explained as one of:
- authoritative state
- app command
- use-case handler
- adapter
- UI view-model
- render projection

then it is probably architectural noise and should not be introduced.
