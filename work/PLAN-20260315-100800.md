# Summary of feature

Fix the remaining architectural problems in neozeus without regressing terminal correctness, redraw correctness, multi-terminal behavior, or the current performance wins.

Target end-state:
- terminal **domain** is separate from Bevy **presentation**
- logical terminal creation is separate from Bevy entity/image creation
- terminal rasterization remains outside Bevy text rendering
- latest-frame mailbox remains, but as a real abstraction with explicit ownership
- runtime terminal code is separated from verification/test hooks
- stale 3D/"plane" vocabulary and dead fields are removed
- `main.rs` is bootstrap only, not a global namespace / dependency hub
- no crate-wide implicit coupling via `use crate::*`
- no `pub(crate) use *` facade leaking internals

Must not regress:
- terminal text rendering/input
- multi-terminal independence
- pixel-perfect HUD promotion
- Vello HUD/demo
- reactive redraw behavior
- long-output responsiveness

---

# Overview of the steps

1. **Freeze invariants and current ownership**
2. **Stop crate-wide implicit coupling (`use crate::*`)**
3. **Replace `src/terminals.rs` glob facade with narrow exports**
4. **Split logical terminal creation from Bevy presentation creation**
5. **Split domain types from Bevy-facing types**
6. **Delete the dead helper-font entity path**
7. **Remove stale `plane` / 3D vocabulary and dead fields**
8. **Encapsulate terminal update mailbox + wake protocol**
9. **Split terminal manager / bridge / runtime / debug responsibilities**
10. **Separate status / error / lifecycle semantics**
11. **Move auto-verify/test plumbing out of terminal runtime bootstrap**
12. **Reduce orchestration coupling in `scene.rs`**
13. **De-god `main.rs` and central app constants**
14. **DRY pass on HUD / presentation / debug glue**
15. **Validation, profiling, cleanup**

# Details of execution

## Phase 0 — Freeze invariants and current ownership

### Goal
Refactor without accidentally changing the parts that are already correct.

### Changes
Write down the ownership model that the code will converge to:
- `terminal domain`
  - PTY worker
  - terminal updates / damage / lifecycle
  - mailbox
- `terminal raster`
  - font discovery
  - glyph cache
  - surface -> RGBA composition
- `terminal presentation`
  - Bevy image assets
  - panel entities / HUD surface
  - transform/interpolation/redraw scheduling
- `ui / verification`
  - egui toolbar
  - optional automation / GUI verification

### Invariants
- terminal rasterization stays outside Bevy text rendering
- Bevy stays host/presenter/compositor
- terminal focus remains separate from terminal position/presentation
- mailbox remains latest-only for visual frames
- pixel-perfect HUD remains optional and active-terminal scoped

### Traps
- refactor work can silently reintroduce Bevy-text dependence
- refactor work can silently merge domain and presentation again if spawn/presentation ownership is not fixed early

### Phase gate
- no code changes required
- use this phase as the checklist for all later phases

## Phase 1 — Stop crate-wide implicit coupling

### Goal
Make dependencies explicit. Current module split is weakened by `use crate::*` everywhere.

### Changes
Replace `use crate::*` in submodules with explicit imports.

Representative change:
```rust
// before
use crate::*;

// after
use crate::{DEFAULT_BG, DEFAULT_CELL_WIDTH_PX, DEFAULT_CELL_HEIGHT_PX};
use crate::terminals::{TerminalManager, TerminalSurface, TerminalDamage};
```

### Execution
- start with `src/terminals/*`
- then `scene.rs`, `input.rs`, `hud.rs`
- keep broad external dependency imports localized

### Invariants
- behavior unchanged
- compile failures caused by explicit imports are treated as architecture findings

### Traps
- do not replace `use crate::*` with another local pseudo-glob/prelude module; that just moves the smell

### Phase gate
- `cargo check`
- `cargo test`
- `cargo clippy --all-targets -- -D warnings`

## Phase 2 — Replace `src/terminals.rs` glob facade with narrow exports

### Goal
Stop leaking terminal internals across the whole app.

### Current issue
`src/terminals.rs` re-exports everything from every terminal submodule. Internal helpers stay globally reachable.

### Changes
Export only the real cross-module API.

Representative shape:
```rust
pub(crate) use backend::{build_surface, compute_terminal_damage, terminal_worker};
pub(crate) use fonts::{configure_terminal_fonts, resolve_terminal_font_report};
pub(crate) use registry::{TerminalManager, TerminalId, ManagedTerminal};
pub(crate) use mailbox::{TerminalUpdateMailbox, DrainedTerminalUpdates};
pub(crate) use presentation::{sync_terminal_presentations, sync_terminal_hud_surface};
pub(crate) use raster::{create_terminal_image, sync_terminal_texture};
pub(crate) use types::{TerminalSurface, TerminalUpdate, TerminalCommand, TerminalDamage};
```

### Invariants
- internal helpers remain internal
- compile errors are used to discover wrong layer crossings

### Traps
- do not respond to missing exports by making everything `pub(crate)` again

### Phase gate
- `cargo check`
- `cargo test`

## Phase 3 — Split logical terminal creation from Bevy presentation creation

### Goal
Remove the biggest current separation-of-concerns violation: `TerminalManager` creates Bevy images/entities directly.

### Current issue
`TerminalManager::spawn_terminal(&mut Commands, &mut Assets<Image>, ...)` means the domain/registry layer owns presenter construction.

### Changes
Split into:
- logical terminal creation in registry/runtime
- presentation creation in scene/presentation layer

Representative shape:
```rust
// registry/domain
pub(crate) fn create_terminal(&mut self, auto_verify: bool) -> Result<TerminalId, String>;

// presentation/scene
pub(crate) fn spawn_terminal_presentation(
    commands: &mut Commands,
    images: &mut Assets<Image>,
    terminal_id: TerminalId,
) -> TerminalPresentationHandles;
```

Or with a small return object:
```rust
pub(crate) struct CreatedTerminal {
    pub(crate) id: TerminalId,
    pub(crate) bridge: TerminalBridge,
}
```

### Execution
- registry creates `ManagedTerminal` with logical/runtime state only
- presentation layer attaches image handle / panel entities afterward
- manager stores logical terminal state; presenter stores entity/image mapping

### Invariants
- logical terminal existence must not imply a specific sprite/image implementation
- multi-terminal IDs/focus/order semantics stay unchanged

### Traps
- if this phase is delayed, later cleanup keeps fighting the same domain/presentation coupling
- do not move the same coupling into another method with a different name

### Phase gate
- `cargo check`
- live smoke: spawn terminal via HUD still works

## Phase 4 — Split domain types from Bevy-facing types

### Goal
Separate terminal-domain data from Bevy resources/components.

### Current issue
`state.rs` mixes:
- pure domain types: `TerminalSurface`, `TerminalCell`, `TerminalDamage`, `TerminalUpdate`
- Bevy state/types: `TerminalPlaneState`, `TerminalGlyphCache`, `TerminalHudSurfaceMarker`, etc.

### Changes
Split types into domain vs Bevy-facing modules.

Recommended shape:
```text
terminals/
  types.rs            # TerminalSurface, TerminalCell, TerminalUpdate, TerminalDamage
  runtime_state.rs    # bridge/runtime mailbox-facing state
  presentation_state.rs  # Bevy resources/components/texture state
```

Representative split:
```rust
// types.rs
pub(crate) struct TerminalSurface { ... }
pub(crate) enum TerminalUpdate { ... }

// presentation_state.rs
#[derive(Resource)]
pub(crate) struct TerminalViewState { ... }
#[derive(Component)]
pub(crate) struct TerminalPanel { ... }
```

### Invariants
- domain types compile without Bevy-specific baggage where possible
- presentation state remains Bevy-facing

### Traps
- avoid a fake split where files move but dependencies still flow both ways through broad imports

### Phase gate
- `cargo check`
- `cargo test`

## Phase 5 — Delete the dead helper-font entity path

### Goal
Remove leftover Bevy-text scaffolding from the active terminal renderer.

### Current issue
Active renderer uses `cosmic-text`, but still carries:
- `TerminalFontEntities`
- helper text entities in scene startup
- `sync_terminal_font_helpers(...)`
- `helper_entities` in terminal state
- `select_terminal_font_role(...) -> (TerminalFontRole, Entity, bool)` while the `Entity` is unused by rasterization

### Changes
- delete helper font entities and their sync system
- remove `helper_entities` from manager/texture state
- reduce font-role selection to only what the rasterizer needs
- re-evaluate whether Bevy `Assets<Font>` are needed at all after this deletion

Representative change:
```rust
// before
fn select_terminal_font_role(...) -> (TerminalFontRole, Entity, bool)

// after
fn select_terminal_font_role(...) -> (TerminalFontRole, bool)
```

### Decision point
After deletion, choose explicitly:
1. **Remove Bevy `Font` asset loading entirely** if it no longer serves any active path.
2. **Keep it only if a remaining Bevy-facing use is real and documented.**

### Invariants
- font fallback behavior remains identical
- emoji/private-use routing remains intact
- no Bevy text rendering is reintroduced

### Traps
- some tests may indirectly rely on current font asset behavior; keep only the minimum real requirement

### Phase gate
- `cargo check`
- `cargo test`
- live smoke: terminal glyphs still render correctly

## Phase 6 — Remove stale `plane` / 3D vocabulary and dead fields

### Goal
Rename the presenter to match what it actually is now.

### Current issue
The app is fundamentally 2D panel + HUD presentation, but still uses stale names:
- `TerminalPlaneState`
- `TerminalPlaneMarker`
- `drag_terminal_plane`
- `zoom_terminal_plane`
- `sync_terminal_plane_transform`
- dead `yaw`, `pitch`, `focal_length`

### Changes
Rename to presenter/view terminology and delete dead state.

Recommended rename set:
- `TerminalPlaneState` -> `TerminalViewState`
- `TerminalPlaneMarker` -> `TerminalPanelSprite`
- `drag_terminal_plane` -> `drag_terminal_view`
- `zoom_terminal_plane` -> `zoom_terminal_view`
- `sync_terminal_plane_transform` -> `sync_terminal_presentations`

Delete dead fields:
```rust
pub(crate) struct TerminalViewState {
    pub(crate) distance: f32,
    pub(crate) offset: Vec2,
}
```

### Invariants
- interaction behavior unchanged
- presentation behavior unchanged

### Traps
- rename atomically across input/presentation/HUD; partial rename leaves the code semantically worse

### Phase gate
- `cargo check`
- live smoke: pan/zoom/scrollback still behave the same

## Phase 7 — Encapsulate terminal update mailbox + wake protocol

### Goal
Keep latest-frame coalescing, but stop representing it as a protocol spread across free functions and shared mutable state.

### Current issue
Current behavior is split across:
- `PendingTerminalUpdates`
- `record_terminal_update(...)`
- `enqueue_terminal_update(...)`
- `drain_terminal_updates(...)`
- `wake_pending` contract

Correct, but too implicit.

### Changes
Introduce a dedicated mailbox type with a narrow API.

Representative shape:
```rust
pub(crate) struct TerminalUpdateMailbox {
    inner: Mutex<PendingTerminalUpdates>,
}

impl TerminalUpdateMailbox {
    pub(crate) fn push_frame(&self, frame: TerminalFrameUpdate) -> MailboxPush;
    pub(crate) fn push_status(&self, status: LatestTerminalStatus) -> MailboxPush;
    pub(crate) fn drain(&self) -> DrainedTerminalUpdates;
}

pub(crate) struct MailboxPush {
    pub(crate) should_wake: bool,
}
```

### Ownership rule
- mailbox owns latest-frame storage and wake coalescing only
- stats accounting happens outside the mailbox or in a thin bridge wrapper, explicitly

### Invariants
- latest visual frame wins
- status updates are not lost
- redraw wakeups remain coalesced

### Traps
- do not count dropped frames in both mailbox and drain path
- keep the mailbox independent from Bevy except for wake signaling wrapper at the edge

### Phase gate
- `cargo test`
- add mailbox unit tests for:
  - latest frame wins
  - status survives with frame
  - wake only requested once until drained

## Phase 8 — Split terminal manager / bridge / runtime / debug responsibilities

### Goal
Stop `manager.rs` from being a dumping ground.

### Current issue
Current `manager.rs` mixes:
- logical terminal registry/focus/order
- worker/bridge bootstrap
- mailbox helpers
- debug logging/stats
- auto-verify dispatch
- status/error helpers

### Changes
Split responsibilities explicitly:
- `registry.rs`
  - `TerminalManager`, `ManagedTerminal`, focus/order/logical terminal ownership
- `bridge.rs`
  - `TerminalBridge`, UI-facing command send / stats snapshot interface
- `runtime.rs`
  - worker thread bootstrap, panic forwarding, notifier wiring
- `mailbox.rs`
  - update mailbox abstraction
- `debug.rs`
  - `TerminalDebugStats`, debug logging helpers

Representative shape:
```rust
// registry.rs
pub(crate) struct TerminalManager { ... }

// bridge.rs
pub(crate) struct TerminalBridge { ... }

// runtime.rs
pub(crate) fn spawn_terminal_runtime(...) -> TerminalBridge
```

### Invariants
- `TerminalManager` remains the UI-facing owner of logical terminals
- bridge remains the only path for UI -> terminal commands

### Traps
- do not create many tiny files with no clear ownership benefit
- `spawn_terminal_runtime(...)` must not grow into another god-function

### Phase gate
- `cargo check`
- `cargo test`

## Phase 9 — Separate status / error / lifecycle semantics

### Goal
Stop collapsing unrelated runtime meanings into a single status+error channel.

### Current issue
`send_terminal_status_update(...)` also mutates error state. Normal lifecycle notices get treated like errors.

### Changes
Introduce explicit runtime semantics.

Recommended shape:
```rust
pub(crate) enum TerminalLifecycle {
    Running,
    Exited { code: Option<i32>, signal: Option<i32> },
    Disconnected,
    Failed(String),
}

pub(crate) struct TerminalRuntimeState {
    pub(crate) status: String,
    pub(crate) lifecycle: TerminalLifecycle,
    pub(crate) last_error: Option<String>,
}
```

Minimal acceptable version:
- separate `send_terminal_status_update(...)`
- separate `send_terminal_error_update(...)`
- explicit exit/disconnect notices without forcing `last_error`

### Invariants
- real failures still surface clearly in HUD/debug state
- normal exits/disconnects are not mislabeled as errors unless they are errors

### Traps
- existing HUD/debug display may assume `last_error` is always populated on shutdown; update that display logic intentionally

### Phase gate
- `cargo test`
- live smoke: intentional terminal exit is reported sanely

## Phase 10 — Move auto-verify/test plumbing out of terminal runtime bootstrap

### Goal
Remove verification hooks from the normal runtime terminal path.

### Current issue
`spawn_auto_verify_dispatcher(...)` is attached to terminal bridge spawn. Runtime terminal bootstrap knows about test-driving commands.

### Changes
Move auto-verify into optional app/plugin/bootstrap layer.

Recommended shape:
- `verification/runtime.rs`
- `AutoVerifyConfig::from_env()` resource
- startup system installs dispatcher only when config exists
- terminal runtime stays unaware

Representative wiring:
```rust
if let Some(config) = AutoVerifyConfig::from_env() {
    app.insert_resource(config)
        .add_systems(Startup, schedule_auto_verify_for_primary_terminal);
}
```

### Invariants
- GUI verification scripts still work
- normal terminal spawn path has zero verification-specific code

### Traps
- auto-verify must wait until the primary logical terminal and its bridge exist

### Phase gate
- GUI suite still passes when intentionally run

## Phase 11 — Reduce orchestration coupling in `scene.rs`

### Goal
Make runtime ordering explicit without serializing everything through one blunt chain.

### Current issue
`scene.rs` chains a long `Update` pipeline. Easy to reason about, but too coarse and too coupled.

### Changes
Use named sets with explicit ordering edges.

Representative shape:
```rust
#[derive(SystemSet, Debug, Clone, PartialEq, Eq, Hash)]
pub(crate) enum NeoZeusSet {
    PollTerminal,
    RasterTerminal,
    PresentTerminal,
    UiInput,
    Redraw,
}
```

### Required ordering edges
- mailbox poll/apply before raster
- raster before presentation sync
- presentation sync before redraw request
- input independent unless it mutates state consumed in the same frame

### Suggested placement
- `poll_terminal_snapshots` -> `PollTerminal`
- `sync_terminal_texture` -> `RasterTerminal`
- `sync_terminal_presentations`, `sync_terminal_panel_frames`, `sync_terminal_hud_surface` -> `PresentTerminal`
- `forward_keyboard_input`, pan/zoom/scrollback input -> `UiInput`
- `request_redraw_while_visuals_active`, `sync_eva_vector_demo` -> `Redraw` or post-present edge as needed

### Invariants
- redraw scheduler sees final visual state of the frame
- no one-frame lag introduced by bad set ordering

### Traps
- Bevy schedule changes can create subtle one-frame regressions; verify live typing and Vello animation after each scheduling pass

### Phase gate
- live smoke: typing repaints without incidental input
- live smoke: Vello animation stays active under reactive mode

## Phase 12 — De-god `main.rs` and central app constants

### Goal
Make `main.rs` bootstrap-only instead of the global namespace / dependency hub.

### Current issue
Even after explicit imports, `main.rs` still owns too many constants and broad dependency imports.

### Changes
Move shared app constants/config to a dedicated module.

Recommended shape:
```text
src/
  app_config.rs
  main.rs
```

Representative contents:
```rust
pub(crate) const DEFAULT_COLS: u16 = 120;
pub(crate) const DEFAULT_ROWS: u16 = 38;
pub(crate) const DEFAULT_BG: egui::Color32 = ...;
pub(crate) const DEBUG_LOG_PATH: &str = "/tmp/neozeus-debug.log";
```

`main.rs` then becomes:
- bootstrap
- error exit handling
- test module only if still acceptable

### Invariants
- no behavior change
- `main.rs` becomes narrow and boring

### Traps
- do not replace one god-module with another giant `common.rs`; keep config/constants cohesive only

### Phase gate
- `cargo check`
- `cargo test`

## Phase 13 — DRY pass on HUD / presentation / debug glue

### Goal
Reduce repeated formatting and state-target computation without hiding logic.

### Targets
1. **HUD debug formatting**
   - extract formatted debug line helper(s)
2. **Presentation target computation**
   - extract pure `compute_terminal_presentation_target(...)`
3. **Surface sizing helpers**
   - unify HUD/world/frame sizing naming and call sites
4. **Optional debug logging wrapper**
   - centralize env-gated logging behavior

Representative shape:
```rust
struct TerminalPresentationTarget {
    position: Vec2,
    size: Vec2,
    alpha: f32,
    z: f32,
}

fn compute_terminal_presentation_target(...) -> TerminalPresentationTarget
```

### Invariants
- helper extraction only; no behavior changes hidden inside DRY work

### Traps
- do not DRY away clarity; repeated logic should only be extracted when semantics are stable

### Phase gate
- `cargo clippy --all-targets -- -D warnings`

## Phase 14 — Validation, profiling, cleanup

### Required validation after every major phase
- `cargo check`
- `cargo test`
- `cargo clippy --all-targets -- -D warnings`
- `cargo fmt --check`

### Required live checks after phases touching runtime/presentation
- typing in active terminal updates visibly
- pixel-perfect HUD promotion still works
- Vello animation still repaints under reactive mode
- multi-terminal focus routing still works

### Required GUI verification when relevant
- optional GUI suite still passes when intentionally run

### Architecture acceptance criteria
- no `use crate::*` in submodules
- no `pub(crate) use *` terminal facade
- logical terminal creation separated from presentation creation
- domain types separated from Bevy-facing types
- no helper font entities in active renderer path
- no stale `yaw` / `pitch` / `focal_length`
- mailbox is a dedicated abstraction, not a free-function protocol
- status/error/lifecycle semantics separated
- no runtime auto-verify logic inside terminal bridge/bootstrap
- `main.rs` reduced to bootstrap role

### Perf acceptance criteria
- no regression in reactive idle behavior
- no visible typing regression
- no regression in long-output catch-up relative to the latest-frame mailbox baseline
- no increase in avoidable wake churn

### Recommended execution order
1. explicit imports
2. narrow exports
3. split logical terminal creation from presentation creation
4. split domain types from Bevy-facing types
5. delete helper-font entity path
6. remove stale plane vocabulary
7. encapsulate mailbox
8. split manager / bridge / runtime / debug
9. separate status / error / lifecycle
10. move auto-verify out of runtime bootstrap
11. scene scheduling cleanup
12. de-god `main.rs`
13. DRY cleanup
14. final validation

### Why this order
- remove coupling first
- fix the major architectural ownership bug early
- clean out stale renderer leftovers only after ownership is clearer
- encapsulate protocols after their boundaries are explicit
- leave DRY and naming polish until structure is correct
