# Summary of feature

Add durable terminal persistence using **tmux as the process authority** and **neozeus metadata as the UI/domain authority**.

Target end-state:

- new logical terminals spawn into **named detached tmux sessions**, not raw shells
- neozeus attaches a PTY client to each tmux session for rendering/input
- if neozeus crashes or restarts, startup **discovers + reconciles + restores** live tmux-backed sessions
- startup restores terminal UI/domain metadata (order, label, optional focus) from persisted neozeus state
- `Ctrl-K` kills the **active tmux-backed terminal session** and removes it from neozeus
- normal startup remains robust when tmux is unavailable or metadata is stale/corrupt
- no existing HUD/presentation/rendering invariants regress

Non-goals for the first implementation:

- multi-client ownership locking across multiple neozeus instances
- preserving in-flight scrollback/view offset beyond what tmux itself already preserves

---

# Overview of the steps

1. **Introduce tmux session identity + persisted metadata types**
2. **Refactor terminal backend spawn into a configurable session target**
3. **Spawn new terminals into detached tmux sessions and attach neozeus PTY to them**
4. **Add explicit active-terminal kill semantics (`Ctrl-K`)**
5. **Persist terminal metadata whenever terminal set/focus/labels change**
6. **Implement startup recovery and reconcile metadata vs live tmux sessions**
7. **Preserve current UX semantics**
   - normal startup: no terminal unless recovered/imported
   - plain `z`: create first hidden tmux-backed terminal if none exists
   - focus path still brings a recovered/hidden terminal foreground
   - `Ctrl-K`: kill active tmux session + remove it from neozeus
8. **Add deterministic tests + GUI verification compatibility**

---

# Details of execution

## 1) Add durable session metadata layer

### Goal

Persist neozeus-side identity for each logical terminal independently from the PTY attachment.

### New module

Create a dedicated persistence module under terminals, not HUD:

```text
src/terminals/
  session_persistence.rs
```

### New types

```rust
#[derive(Clone, Debug, PartialEq, Eq)]
pub(crate) struct TerminalSessionRecord {
    pub(crate) session_name: String,
    pub(crate) label: Option<String>,
    pub(crate) creation_index: u64,
    pub(crate) last_focused: bool,
}

#[derive(Clone, Debug, Default, PartialEq, Eq)]
pub(crate) struct PersistedTerminalSessions {
    pub(crate) sessions: Vec<TerminalSessionRecord>,
}

#[derive(Resource, Default)]
pub(crate) struct TerminalSessionPersistenceState {
    pub(crate) path: Option<PathBuf>,
    pub(crate) dirty_since_secs: Option<f32>,
}
```

### Path

Use XDG/Home like HUD persistence, but **state-oriented** rather than layout-oriented. Prefer:

- `${XDG_STATE_HOME}/neozeus/terminals.v1` if added
- else `${HOME}/.local/state/neozeus/terminals.v1`
- fallback to `${XDG_CONFIG_HOME}` / `${HOME}/.config` only if keeping consistency is more important than purity

### Format

Keep the current project style: line-based, versioned, cheap to diff.

Example:

```text
version 1
session name=neozeus-20260316-abc label=agent-1 creation_index=0 focused=1
session name=neozeus-20260316-def creation_index=1 focused=0
```

### Invariants

- persisted metadata never becomes the authority for process continuity; tmux is
- metadata file is best-effort; load/save failure must not abort startup
- session names are stable and unique; `TerminalId` is not persisted as durable identity

### Traps

- do not key persistence off `TerminalId`; it is per-process and non-durable
- do not store transient attachment state in persistence
- do not couple terminal metadata persistence to HUD layout persistence

## 2) Introduce explicit terminal session target abstraction

### Goal

Separate “what process/session to attach to” from raw PTY mechanics.

### Current problematic point

`backend::spawn_pty()` hardcodes shell spawn:

```rust
let shell = shell_path();
let mut command = CommandBuilder::new(shell);
command.env("TERM", "xterm-256color");
```

This must become parameterized.

### New types

```rust
pub(crate) enum TerminalAttachTarget {
    RawShell,
    TmuxAttach { session_name: String },
}

pub(crate) enum TerminalProvisionTarget {
    RawShell,
    TmuxDetached { session_name: String },
}
```

`ProvisionTarget` = what to create before attaching.
`AttachTarget` = what PTY child neozeus itself runs.

### Refactor shape

- keep `portable_pty` + `alacritty_terminal` stack intact
- replace `spawn_pty(cols, rows)` with something like:

```rust
fn spawn_pty(cols: u16, rows: u16, target: TerminalAttachTarget) -> Result<PtySession, String>
```

and a command builder helper:

```rust
fn build_attach_command(target: &TerminalAttachTarget) -> CommandBuilder
```

### Important attach command

For tmux-backed terminals, neozeus PTY should run:

```text
tmux attach-session -t <session_name>
```

### Invariants

- terminal rendering/input pipeline above PTY spawn remains unchanged
- bridge/mailbox/runtime notifier interfaces stay stable if possible
- `alacritty_terminal` still parses the attached PTY stream the same way

### Traps

- `tmux new-session` and `tmux attach-session` are different concerns; do not conflate them in one spawn path
- if attach fails, status must surface as terminal runtime failure, not silent empty window

## 3) Add tmux session provisioning for new logical terminals

### Goal

New terminals create a durable tmux session first, then attach neozeus to it.

### New helper(s)

Create a small tmux integration helper module under terminals:

```text
src/terminals/tmux.rs
```

Functions:

```rust
pub(crate) fn ensure_tmux_available() -> Result<(), String>;
pub(crate) fn create_detached_session(session_name: &str) -> Result<(), String>;
pub(crate) fn list_sessions() -> Result<Vec<String>, String>;
pub(crate) fn session_exists(session_name: &str) -> Result<bool, String>;
```

### Session naming

Generate names independently of `TerminalId`.
Recommended format:

```text
neozeus-<unixsecs>-<counter or random suffix>
```

Need collision-safe generation:

- check with `tmux has-session`
- retry on collision

### Spawn path for a new terminal

Current spawn path:

- runtime spawner creates PTY runtime
- manager creates terminal
- presentation spawn

New path:

1. generate session name
2. `tmux new-session -d -s <name>`
3. runtime spawner attaches via `tmux attach-session -t <name>`
4. manager creates logical terminal
5. manager/session registry associates terminal with session name
6. presentation spawn
7. metadata marked dirty

### Needed manager extension

`TerminalManager` currently stores only `bridge/snapshot/damage/revision`.
Add session identity:

```rust
pub(crate) struct ManagedTerminal {
    pub(crate) bridge: TerminalBridge,
    pub(crate) session_name: String,
    ...
}
```

### Invariants

- a logical terminal always has exactly one session name
- session name is available from manager without reading persistence file
- normal app startup must still allow zero terminals

### Traps

- if detached session creation succeeds but attach fails, you now have an orphan tmux session; decide policy upfront:
  - first implementation: keep it, log it, let startup recovery import it later
- verifier mode likely still needs deterministic startup; preserve verifier flow explicitly

## 4) Add explicit active-terminal kill semantics (`Ctrl-K`)

### Goal

Allow the user to destroy the active tmux-backed session intentionally and remove its logical terminal from neozeus.

### Semantics

- `Ctrl-K` targets the **active terminal only**
- if no terminal is active: no-op
- action is **destructive**: kill tmux session, detach/remove neozeus terminal, update metadata
- if killed terminal was active, resulting focus defaults to **none** for the first implementation

### Recommended command shape

Add an explicit command rather than burying kill logic in input handling:

```rust
pub(crate) enum HudCommand {
    ...
    KillActiveTerminal,
}
```

Or, if keeping it outside HUD, add a terminal action routed from `input.rs` into a small dispatcher/resource. The important part is to keep the destructive operation centralized.

### Runtime behavior

1. resolve active terminal from `TerminalManager`
2. read its `session_name`
3. issue `tmux kill-session -t <session_name>`
4. remove terminal from runtime/domain/presentation state
5. remove label metadata if present
6. mark session persistence dirty

### Needed manager/presentation changes

Current code has no terminal removal lifecycle. This feature requires it.

Add at minimum:

```rust
impl TerminalManager {
    pub(crate) fn remove_terminal(&mut self, id: TerminalId) -> Option<ManagedTerminal>;
}

impl TerminalPresentationStore {
    pub(crate) fn remove(&mut self, id: TerminalId) -> Option<PresentedTerminal>;
}
```

And a scene-level cleanup path for `TerminalPanel` / `TerminalPanelFrame` entities owned by that `TerminalId`.

### Invariants

- kill path removes all runtime/domain/presentation references for that terminal
- no stale `active_id`, `focus_order`, `creation_order`, HUD label, or presentation entry remains
- failure to kill tmux session must surface clearly; do not silently drop the terminal locally unless policy explicitly says so

### Traps

- kill is the first true terminal removal path; stale entity/resource cleanup is the predictable bug source
- do not leave orphan sprite/frame entities for removed terminal IDs
- decide policy for tmux kill failure before coding:
  - recommended: if `tmux kill-session` fails, keep terminal alive and report/log failure

## 5) Persist terminal metadata on mutations

### Goal

Keep enough UI/domain state to reconstruct neozeus after restart.

### Mutations that should mark persistence dirty

- new terminal created
- terminal focused
- active terminal killed/removed
- agent label changed
- terminal ordering changes (today: creation order append only, focus changes tracked separately)

### Save system

Add a save system parallel to HUD layout persistence:

```rust
save_terminal_sessions_if_dirty(
    time,
    terminal_manager,
    agent_directory,
    terminal_session_persistence_state,
)
```

Persist from **current resources**, not from stale cached records.

### Metadata source of truth during runtime

- live session names/order/focus: `TerminalManager`
- labels: `AgentDirectory`

### Invariants

- persisted order must reflect `TerminalManager.creation_order`, not focus order
- persisted focus must reflect `active_id`
- save failures log + clear debounce safely; no panic

### Traps

- avoid persisting via HUD code paths; this is terminal/domain persistence
- do not silently lose metadata because focus changed but “terminal set didn’t”

## 6) Implement startup recovery / reconcile

### Goal

On startup, recover live tmux-backed sessions and restore neozeus state.

### Startup flow change

Current normal startup creates no terminal unless verifier mode injects one.
New startup flow:

1. load persisted terminal metadata
2. list live tmux sessions
3. reconcile
4. for each recovered/imported session:
   - spawn runtime attached to session
   - create logical terminal **without implicit focus**
   - spawn presentation
   - optionally apply label metadata
5. restore active/focused session if present and still live
6. verifier mode remains special-case and should not duplicate recovered sessions

### Reconcile algorithm

Inputs:

- `persisted.sessions`
- `tmux list-sessions`

Output categories:

- **restore**: in metadata and live in tmux
- **prune**: in metadata but missing in tmux
- **import**: live tmux session with `neozeus-` prefix but no metadata

Recommended behavior:

- restore all `restore`
- log + drop `prune`
- import `import` with default label/order appended after restored ones

### Key snippet

```rust
for record in reconciled.restore_then_import_order() {
    let bridge = runtime_spawner.spawn_attached(TerminalAttachTarget::TmuxAttach {
        session_name: record.session_name.clone(),
    });
    let (terminal_id, slot) = terminal_manager.create_terminal_without_focus_with_slot_and_session(
        bridge,
        record.session_name.clone(),
    );
    spawn_terminal_presentation(..., terminal_id, slot);
    if let Some(label) = record.label.clone() {
        agent_directory.labels.insert(terminal_id, label);
    }
}
```

Then restore focus after all sessions are created.

### Where to hook it

`scene::setup_scene()` is the right place; it already owns initial terminal creation.

### Invariants

- startup recovery must not create duplicate sessions
- recovery order is deterministic
- if nothing recoverable exists, app still starts empty
- verifier mode must remain reliable in CI/tooling

### Traps

- if verifier mode is enabled and recovery also finds sessions, decide precedence explicitly
  - recommended: verifier mode bypasses recovery and spawns its own deterministic session
- importing arbitrary tmux sessions is dangerous; only import `neozeus-*` prefixed sessions

## 7) Preserve current UX semantics while switching backend

### Goal

Do not regress the currently implemented “empty startup + plain z bootstrap + hidden until focused” behavior.

### Required semantics

- normal startup: no terminal unless recovered/imported
- plain `z` when no terminals exist: create a **detached tmux session** + hidden neozeus attachment, unfocused
- existing focus path (currently AgentList row click) brings terminal foreground
- `Ctrl-K` kills the active session and leaves no active terminal unless product later chooses fallback focus

### Effect on current code

The recent bootstrap path should stop creating raw-shell terminals and instead provision tmux-backed ones.

### Invariants

- `z` shortcut semantics unchanged from the user’s perspective
- hidden/unfocused terminal remains hidden until focus
- `Ctrl-K` is destructive only for the active terminal and is a no-op with no active terminal
- no background terminal should render when `active_id` is `None`

### Traps

- if recovery restores one or more sessions, `z` bootstrap should no longer trigger because terminal count is non-zero
- imported sessions with no focus metadata should probably start with no active terminal unless product decision says otherwise

## 11) Critical invariants to maintain throughout

- no raw startup terminal in normal mode unless recovery/import restores one
- verifier mode remains deterministic and non-flaky
- PTY render/input pipeline above attach target remains unchanged
- terminal creation order remains stable across focus changes
- terminal removal leaves no stale manager/store/entity references
- hidden terminal remains hidden while `active_id == None`
- metadata corruption or tmux failure must degrade gracefully, not abort startup
- no new external dependency should be introduced; use system tmux executable

## 12) Predictable traps to address before coding

### Trap: tmux missing on host

Handle explicitly at provision/attach time with user-visible runtime status/logging.
Do not panic. Likely behavior:

- new terminal creation fails with log + no terminal inserted
- recovery skips with clear status/log

### Trap: verifier path regression

The current GUI suite depends on a deterministic startup terminal. Preserve this via explicit verifier-only startup spawn and keep it outside normal recovery path.

### Trap: duplicate restore/import

If metadata says session exists and live tmux listing also sees it, restore exactly once.

### Trap: stale metadata after external tmux kill

Startup reconcile must prune dead records rather than recreating dead logical terminals.

### Trap: command quoting for attach/session creation

Avoid spawning through shell when not needed. Prefer direct argv to `tmux` subcommands using `CommandBuilder`/`std::process::Command`.

### Trap: removal path fan-out

Killing a terminal now requires coherent cleanup across:
- tmux session
- `TerminalManager`
- `TerminalPresentationStore`
- terminal panel/frame entities
- `AgentDirectory`
- persisted session metadata

Miss one and the app will drift into stale state.

### Trap: using persisted focus blindly

If persisted focused session is missing, no fallback silent focus should occur unless product explicitly wants “first restored terminal becomes active”. Default safer behavior: restore persisted focus only if live; else no active terminal.
