# Summary of feature

Create a staged implementation plan to move neozeus from its current CPU-composited terminal texture pipeline to kitty-class responsiveness while preserving the current architectural constraints:

- terminal semantics stay on `alacritty_terminal + portable-pty`
- active text rasterization stays outside Bevy’s text pipeline
- Bevy remains scene/compositor/presenter
- current features must keep working: multi-terminal management, pixel-perfect HUD promotion, Vello HUD/demo, optional GUI verification, reactive idle behavior

The parity path is not one change. It is a sequence of increasingly expensive optimizations:

1. remove avoidable queue/snapshot latency
2. remove redundant damage discovery
3. reduce per-cell/per-frame allocations and comparisons
4. make CPU compositor much cheaper
5. stop re-uploading whole textures
6. if still needed, replace CPU RGBA composition with a real GPU glyph-atlas renderer

---

# Overview of the steps

1. **[x] Stabilize performance measurement**
   - added repeatable bulk-output repro + perf scripts
   - added counters for dropped updates / dirty rows / compose cost in HUD/debug stats

2. **[x] Apply newest-only snapshot consumption**
   - main thread now keeps only the newest pending frame update per terminal per frame

3. **[x] Push damage tracking into the worker**
   - worker now computes row damage and sends it with each frame update
   - main thread no longer re-diffs the full visible surface on normal updates

4. **[x] Replace heap-heavy cell model**
   - hot-path cells now use compact `TerminalCellContent`
   - heap fallback remains only for larger grapheme payloads

5. **[x] Rewrite CPU compositor around spans/rows**
   - compositor now writes directly into row slices / CPU backing buffers
   - old `pixel_bytes_mut` per-pixel path is removed from the active renderer

6. **[x] Switch from full asset refresh semantics to partial GPU texture upload**
   - terminal renderer now keeps a CPU backing buffer and uploads contiguous dirty row bands via `RenderQueue::write_texture`

7. **[x] Re-evaluate parity gap**
   - after phases 1-6, the custom GPU glyph-atlas renderer was **not triggered in this pass**
   - current implementation keeps it as the next escalation only if new profiling or user feedback still shows a hard parity gap

8. **[x] Finish with validation + regression tooling**
   - added perf scripts, expanded unit coverage, ran full lint/test/smoke gates

# Details of execution

## Phase 0 — Measurement first

### Goal
Stop guessing. Every later change must prove where time went.

### Changes
Add a lightweight perf harness for a large deterministic tree-output workload.

Suggested artifacts:
- `scripts/perf/run-terminal-bulk-output.sh`
- `scripts/perf/gen-big-tree.sh`
- optional env-gated timing/counter logging

Suggested counters to expose in HUD/debug state:
- PTY bytes read
- frame updates emitted by worker
- frame updates applied by main thread
- dirty rows per applied update
- CPU compose time per update
- GPU upload bytes / dirty rows uploaded
- dropped updates count

### Key changes
```rust
#[derive(Clone, Default)]
pub(crate) struct TerminalPerfStats {
    pub(crate) updates_emitted: u64,
    pub(crate) updates_applied: u64,
    pub(crate) updates_dropped: u64,
    pub(crate) dirty_rows_uploaded: u64,
    pub(crate) compose_micros: u64,
    pub(crate) upload_micros: u64,
}
```

### Invariants
- no perf logging in the hot path unless explicitly enabled
- no change to terminal behavior

### Traps
- do not log per-byte/per-read/per-row in profiling mode; that invalidates the measurements
- keep benchmark workload local and deterministic

## Phase 1 — Newest-only snapshot application

### Goal
Do not spend a frame applying stale terminal states.

### Current problem
`poll_terminal_snapshots(...)` drains all queued snapshots:
```rust
while let Ok(snapshot) = receiver.try_recv() {
    terminal.latest = snapshot;
}
```
This causes catch-up work under bursty output.

### Change
Drain the queue, keep only the newest update, apply one.

### Key changes
```rust
pub(crate) fn poll_terminal_updates(mut terminal_manager: ResMut<TerminalManager>) {
    for terminal in terminal_manager.terminals.values_mut() {
        let receiver = lock_receiver(&terminal.bridge.update_rx);
        let mut latest = None;
        while let Ok(update) = receiver.try_recv() {
            latest = Some(update);
        }
        if let Some(update) = latest {
            terminal.pending_update = Some(update);
            terminal.bridge.note_update_applied();
        }
    }
}
```

Add a dropped-update counter:
```rust
if latest.is_some() {
    stats.updates_dropped += drained_count.saturating_sub(1) as u64;
}
```

### Invariants
- never drop terminal exit/error/status events
- latest visual state must win
- scrollback/manual scroll events still apply in order

### Traps
- if visual updates and status updates share the same channel, newest-only can accidentally drop important terminal lifecycle messages
- fix this by sending structured updates, not raw snapshots

## Phase 2 — Worker-generated damage metadata

### Goal
Stop rebuilding/diffing twice.

### Current problem
Worker builds a full `TerminalSurface`; main thread then compares old/new surfaces again with `dirty_rows_between(...)`.

### Change
Worker owns damage detection and sends it with the update.

### New update type
```rust
pub(crate) struct TerminalFrameUpdate {
    pub(crate) surface: TerminalSurface,
    pub(crate) damage: TerminalDamage,
    pub(crate) status: String,
    pub(crate) seq: u64,
}

pub(crate) enum TerminalDamage {
    Full,
    Rows(SmallVec<[u16; 64]>),
    Rects(SmallVec<[TerminalRect; 16]>),
}
```

### Execution
- worker keeps previous visible surface
- after `build_surface(...)`, compute damage immediately
- send `{surface, damage}`
- renderer uses provided damage directly
- delete `dirty_rows_between(...)` from hot path

### Key changes
```rust
let next_surface = build_surface(&terminal);
let damage = compute_damage(previous_surface.as_ref(), &next_surface);
let update = TerminalFrameUpdate { surface: next_surface.clone(), damage, status, seq };
previous_surface = Some(next_surface);
```

### Invariants
- cursor-only changes must still produce damage
- resize/clear/alt-screen transitions must force `Full`
- no stale rows after scroll operations

### Traps
- scrollback/scroll-region activity can invalidate most rows even if per-cell diff looks sparse
- alt-screen swaps and full clears must bypass row diff heuristics

## Phase 3 — Replace `String` in hot cells

### Goal
Stop allocating and comparing heap strings for ordinary terminal cells.

### Current problem
`TerminalCell` currently stores:
```rust
struct TerminalCell {
    text: String,
    fg: Color32,
    bg: Color32,
    width: u8,
}
```
Most cells are 1-char ASCII/box-drawing/space, but they still pay heap-heavy semantics.

### Change
Use compact inline cell content and style IDs.

### Proposed model
```rust
#[derive(Clone, Copy, Debug, PartialEq, Eq)]
struct TerminalStyleId(u32);

#[derive(Clone, Debug, PartialEq, Eq)]
enum CellContent {
    Empty,
    Single(char),
    InlineSmall([char; 2], u8),
    Heap(Arc<str>),
}

#[derive(Clone, Debug, PartialEq, Eq)]
struct TerminalCell {
    content: CellContent,
    style: TerminalStyleId,
    width: u8,
}
```

### Execution
- intern fg/bg/style combinations into a style table
- keep common path allocation-free
- heap fallback only for combining sequences / emoji / rare clusters

### Invariants
- preserve wide chars, zero-width joins, emoji, private-use glyphs
- no regression in color correctness
- glyph cache keys still distinguish content/size/font role correctly

### Traps
- Unicode clusters are the main correctness risk
- do not collapse everything to a single `char`; keep explicit fallback path for complex graphemes

## Phase 4 — Rewrite CPU compositor around row slices

### Goal
Make the current CPU texture pipeline as cheap as possible before changing render architecture.

### Current problem
Current path uses many scalar operations:
- `fill_rect(...)` loops per pixel
- `blit_cached_glyph(...)` calls `blend_image_pixel(...)`
- `pixel_bytes_mut(...)` inside tight loops

### Change
Operate on raw row slices with fast paths.

### Execution
1. expose mutable raw image buffer once per repaint
2. precompute row stride
3. implement fast opaque background span fill
4. implement glyph blit over slices, not per-pixel asset lookups
5. keep alpha blending only where actually required

### Key changes
```rust
fn fill_span_rgba(dst: &mut [u8], rgba: [u8; 4]) {
    for pixel in dst.chunks_exact_mut(4) {
        pixel.copy_from_slice(&rgba);
    }
}

fn repaint_row(buffer: &mut [u8], stride: usize, y: usize, row: &[TerminalCell], ...) {
    let row_start = y * row_pixel_height * stride;
    // write spans directly
}
```

### Important fast paths
- background fill is opaque -> overwrite, do not blend
- most glyphs are monochrome masks -> use specialized blend path
- skip rendering entirely for runs of empty/default-background cells

### Invariants
- alpha-correct glyph rendering must stay correct
- no regression in cursor rendering
- pixel-perfect HUD path must still use the same composed output semantics

### Traps
- direct buffer math is easy to get wrong on row boundaries
- test cursor rows, double-width chars, emoji, underline/beam/block cursor overlap

## Phase 5 — Partial GPU texture upload

### Goal
Stop treating every terminal image change as a whole-texture asset refresh.

### Current problem
Even if CPU changes only a few rows, Bevy asset mutation likely still turns into coarse GPU upload behavior.

### Change
Own the GPU texture upload path explicitly and update only dirty rows/rects.

### Execution
1. keep CPU backing buffer in terminal state
2. compose dirty rows into that backing buffer
3. create/own GPU texture resource for terminal surfaces
4. use `wgpu::Queue::write_texture` with row/rect regions
5. sprite/material samples that persistent texture

### Key changes
```rust
queue.write_texture(
    ImageCopyTexture { texture: &gpu_texture, mip_level: 0, origin },
    dirty_bytes,
    ImageDataLayout {
        offset: 0,
        bytes_per_row: Some(dirty_width * 4),
        rows_per_image: Some(dirty_height),
    },
    Extent3d { width: dirty_width, height: dirty_height, depth_or_array_layers: 1 },
);
```

### Files likely affected
- `src/terminals.rs`
- likely new render module, e.g. `src/terminals/render_texture.rs`

### Invariants
- keep Bevy as presenter/compositor
- no reintroduction of Bevy text rendering
- multi-terminal independence preserved

### Traps
- Bevy asset abstractions may not expose fine-grained update control cleanly
- if the asset system fights this, use a custom render resource owned by the terminal subsystem

## Phase 6 — Rework terminal update model around explicit frame updates

### Goal
Separate terminal semantics from rendering payloads cleanly.

### Change
Replace generic `TerminalSnapshot` flow with explicit update types.

### Proposed model
```rust
pub(crate) enum TerminalUpdate {
    Frame(TerminalFrameUpdate),
    Status(String),
    Exited { code: Option<i32>, signal: Option<i32> },
    Fatal(String),
}
```

### Why
This makes newest-only dropping safe:
- drop stale `Frame`
- never drop `Exited` / `Fatal`

### Invariants
- terminal lifecycle is never lost
- frame updates can be coalesced safely

### Traps
- do not keep mixing visual updates and terminal lifecycle in the same replacement semantics

## Phase 7 — Optional but likely required for true parity: custom GPU glyph-atlas renderer

### Goal
If CPU texture composition + partial upload still falls short, stop composing terminal text as a CPU RGBA surface.

### Trigger condition
If phases 1-6 still leave neozeus visibly behind kitty on:
- large tree output
- line flood (`yes`, cargo build output, logs)
- interactive scrolling in large outputs

### Change
Keep terminal state on CPU, but render cells via a custom GPU glyph-atlas pipeline.

### Architecture
- worker/main thread produce compact cell grid + damage
- glyphs cached in atlas textures
- renderer submits quads or instanced runs per damaged region
- background/color/cursor handled in GPU pipeline

### Critical constraint
Do **not** fall back to Bevy text for glyph shaping/rasterization.
Use your own atlas generation/cache; Bevy remains host only.

### Minimal design
```rust
struct GlyphAtlasKey {
    content: CellContent,
    font_role: TerminalFontRole,
    cell_size: UVec2,
    style_flags: u8,
}

struct TerminalRenderRun {
    rect: UVec4,
    atlas_uv: UVec4,
    fg: [u8; 4],
    bg: [u8; 4],
}
```

### Invariants
- preserve current font stack/fallback behavior
- preserve exact ANSI colors and cursor visuals
- preserve pixel-perfect HUD mode

### Traps
- hardest part is atlas invalidation for size/font changes
- emoji/color glyph path may need separate atlas handling
- do not attempt this before compact cells + damage propagation exist

## Phase 8 — Validation and acceptance criteria

### Acceptance targets
For the same workload, compared against kitty on the same machine/session:
- first visible output latency: close enough to be hard to perceive
- progressive fill behavior: comparable cadence
- no long “nothing then catch-up” gaps
- idle CPU unchanged from current reactive behavior
- no regressions in text correctness, colors, cursor, multi-terminal behavior, HUD promotion

### Required tests
#### Functional
- existing unit tests stay green
- GUI output verification stays green
- color verification stays green
- pixel-perfect HUD remains visually correct

#### Stress/perf
- large `eza --tree -a` tree
- `cargo build` noisy output
- `yes | head` and sustained flood
- large scrollback drag/scroll behavior
- multi-terminal simultaneous output

### Recommended benchmark script outputs
- bytes/sec read
- updates/sec emitted
- updates/sec applied
- mean dirty rows/update
- CPU compose ms/update
- GPU upload bytes/update
- dropped frame updates count

### Invariants
- every optimization step must preserve terminal correctness first
- performance work must not break the current modular architecture
- no new dependency without approval

### Traps
- optimizing without a benchmark harness will cause churn and false wins
- visual latency is what matters, not only throughput counters

# Recommended implementation order

1. **Phase 1** newest-only frame application
2. **Phase 6** explicit `TerminalUpdate` split
3. **Phase 2** worker-side damage metadata
4. **Phase 3** compact cell representation
5. **Phase 4** row/slice compositor rewrite
6. **Phase 5** partial GPU texture upload
7. benchmark again
8. **Phase 7** custom GPU glyph-atlas renderer only if still needed

This order maximizes performance gain per unit of complexity and avoids doing a hard renderer rewrite before exhausting the cheaper architectural wins.
